# Releasing (maintainer guide)

How to cut a release of **VPN Up for OpenConnect**, including the title/notes
conventions and the end-to-end checklist. Publishing a GitHub release
automatically updates the Homebrew tap formula (see
[`.github/workflows/release-tap.yml`](.github/workflows/release-tap.yml)), so
the title and notes you set here are what users and search engines see.

## Versioning

[Semantic Versioning](https://semver.org/): `vMAJOR.MINOR.PATCH`.

- **MAJOR** — breaking changes (state layout, removed/renamed commands, security
  defaults that can lock out an existing setup). E.g. `v3.0.0` moved user state
  to `~/.config/vpn-up` and made server identity fail closed.
- **MINOR** — new, backward-compatible features (a new command, a new setting).
- **PATCH** — bug fixes and docs/UX polish with no new surface.

## Title convention

```
vX.Y.Z — <Theme>
```

- **`vX.Y.Z`** — the tag, with the `v` prefix.
- **`—`** — an em dash (not a hyphen), with a space on each side.
- **`<Theme>`** — 2–4 words in Title Case naming the headline of the release.
  Keep it identical to the CHANGELOG subtitle for that version so the tag, the
  CHANGELOG, and the GitHub release all read the same.

The **title stays short and human** — do not stuff keywords into it. Search
terms belong in the release **body** (see the template below), where they read
naturally.

**Examples** (and how they map to the CHANGELOG `### <subtitle>`):

| Release title | CHANGELOG subtitle |
|---|---|
| `v3.6.0 — First-Run Usability & Docs` | First-Run Usability & Docs Update |
| `v3.5.0 — Lifecycle & Maintenance` | Lifecycle & Maintenance Update |
| `v3.4.0 — Service & Notifications` | Service & Notifications Update |
| `v3.0.0 — Security Hardening` | Security Hardening Release |

## Release-notes body template

Open with one plain-language sentence that names the product and what it does —
this is the line that captures search intent and shows up in previews — then
paste the matching CHANGELOG section, then link the full changelog.

```markdown
This release of **VPN Up**, a secure command-line VPN manager for OpenConnect
(Cisco AnyConnect, GlobalProtect, Pulse Secure, Juniper) on macOS and Linux,
<one clause on the headline change>.

## <Theme>

### Added
- ...

### Changed
- ...

### Fixed
- ...

### Security
- ...

---

**Full Changelog**: https://github.com/sorinipate/vpn-up-for-openconnect/blob/main/CHANGELOG.md
```

Only include the `###` subsections that actually apply. Lift the bullets
verbatim from the version's CHANGELOG entry — the CHANGELOG is the source of
truth; the release notes are a copy with an SEO lead sentence on top.

## Checklist

`main` is locked and PR-only, so a release is just a normal docs PR followed by
a tag + GitHub release.

1. **Land the work.** All feature/fix PRs for the version are merged to `main`
   and their entries sit under `## [Unreleased]` in [CHANGELOG.md](CHANGELOG.md).
2. **Promote the changelog.** On a release branch, change `## [Unreleased]` to
   `## [vX.Y.Z] — YYYY-MM-DD` and add the `### <Theme> Update` subtitle line.
   ```bash
   git checkout -b release-vX.Y.Z
   # edit CHANGELOG.md
   git commit -am "Release vX.Y.Z: <theme>"
   git push -u origin release-vX.Y.Z
   ```
3. **PR → green → merge.** Open the PR, wait for CI (shellcheck + bats on macOS
   and Ubuntu + gitleaks), then merge. `main` enforces admins, so the merge
   uses a temporary unlock/relock of branch protection — keep that window small.
4. **Tag the merge commit and publish the release.** The tag must exist before
   the release; publishing it triggers the tap-formula update.
   ```bash
   git fetch origin main
   git tag vX.Y.Z <merge-commit-sha>
   git push origin vX.Y.Z
   gh release create vX.Y.Z --target <merge-commit-sha> \
     --title "vX.Y.Z — <Theme>" --notes-file release-notes.md
   ```
5. **Verify the tap.** Confirm the `release-tap.yml` run succeeds and that the
   `homebrew-vpn-up` formula's `url` + `sha256` now point at the new tag. A
   clean `brew update && brew upgrade vpn-up` should pick it up.

> No AI-attribution trailers or "generated by" footers in commits, PRs, tags, or
> release notes.
